 API SECURITY MODEL — SECURE vs INSECURE

        

          

            

              

                

                
JWT STRUCTURE

                
    
                  
  • Header: algorithm type (HS256)
    • 
                  
  • Payload: claims (sub, role, exp, iat)
    • 
                  
  • Signature: HMAC-SHA256(header.payload, secret)
    • 
                  
  • Stateless — no DB lookup per request
    • 
                  
  • Expires automatically (1 hour)
    • 
                

              

            

            

              

                 INSECURE API PATTERNS
                
    
                  
  • No authentication on endpoints
    • 
                  
  • Hardcoded bypass tokens ("admin", "password")
    • 
                  
  • No token expiry — valid forever
    • 
                  
  • No signature verification — any JWT accepted
    • 
                  
  • Verbose error messages leak server info
    • 
                  
  • No rate limiting — unlimited requests
    • 
                  
  • IDOR — predict other user IDs
    • 
                

              

            

            

              

                 SECURE API PATTERNS
                
    
                  
  • Bearer JWT in Authorization header
    • 
                  
  • Short expiry (1 hour max)
    • 
                  
  • Role/permission claims verified
    • 
                  
  • HTTPS only — no plaintext
    • 
                  
  • Rate limiting per token
    • 
                  
  • Minimal error exposure to client
    • 
                  
  • Token blacklist on logout
    • 
                

              

            

          


          

            
 REAL-WORLD JWT VULNERABILITIES 

            

              
alg:none attack: Some libraries accept {"alg":"none"} in the header, skipping signature verification entirely. A forged token with any payload gets accepted.

              
Algorithm confusion (RS256→HS256): If a server accepts both RS256 and HS256, an attacker can forge tokens using the public key as the HMAC secret.

              
Weak secret: JWT signed with "secret" or "password" can be brute-forced offline using tools like hashcat or jwt_tool.

              
Real examples: Auth0 (2015) — alg:none bypass. Portswigger research exposed algorithm confusion attacks in many JWT libraries.
