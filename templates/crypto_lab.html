WHY DOES PASSWORD STRENGTH MATTER?

What is happening: We check 4 criteria: length, uppercase, numbers, special chars. Each adds entropy — the unpredictability of the password.

Why dangerous: Weak passwords are cracked in seconds using dictionary or brute-force attacks. MD5 of "password" is cracked instantly via rainbow tables.

Real-world impact: 81% of data breaches involve weak or stolen passwords (Verizon DBIR). A 6-char lowercase password has only 308 million combinations — cracked in <1 second by modern GPUs.

Best practice: Use a passphrase of 4+ random words, or a password manager generating 20+ char random strings.

HASHING vs ENCRYPTION

Hashing is one-way — you cannot reverse it. Used for passwords. bcrypt is slow by design to resist brute force.

Encryption is two-way — reversible with a key. Used for data at rest/transit (AES, RSA, TLS).

Why bcrypt over SHA-256? SHA-256 hashes 10 billion passwords/second on a GPU. bcrypt limits to ~100/second due to its work factor.

WHY CAESAR CIPHER IS BROKEN

What is happening: Each letter shifts by a fixed number in the alphabet (A→D for shift 3). ROT-13 is a special case with shift 13.

Why dangerous: Only 25 possible shifts — brute force takes <1 second. Frequency analysis (letter 'e' is most common in English) breaks it without knowing the key.

Real-world impact: ROT-13 is still used as "obscurity" in some contexts — completely broken. Modern ciphers (AES-256) use 2^256 possible keys.

BASE64 IN SECURITY CONTEXT

What is it: Base64 converts binary data to printable ASCII using 64 characters. Adds ~33% size. Used in JWT headers/payloads, email attachments, data URIs.

JWT uses Base64URL: The first two parts of a JWT are Base64URL-encoded (not encrypted). Anyone can decode and read the payload — the signature only verifies integrity.

Common mistake: Storing secrets or sensitive data in Base64 thinking it's encrypted. It is not — decode with one command.

HOW SALTING DEFEATS RAINBOW TABLES

The problem: Same input always produces same MD5/SHA hash. Pre-compute all common passwords once → crack any database instantly.

The fix — Salting: Add a random value (salt) before hashing. hash(salt + password) — every hash is unique even for identical passwords. Rainbow tables become useless.

bcrypt automatically salts: The random salt is embedded in the hash string itself (the $2b$10$[22 chars] prefix). This is why two bcrypt hashes of "admin123" look completely different.
