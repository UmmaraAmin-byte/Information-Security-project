LEARNING PANEL
WHAT IS SESSION HIJACKING?

What is happening: After login, the server gives the browser a session token (cookie). This token proves identity for all future requests. If an attacker steals this token, they can impersonate the victim.

Attack vectors: XSS injects JavaScript that reads document.cookie. Network sniffing on HTTP (not HTTPS). Session fixation — attacker sets the session ID before login.

Why HttpOnly matters: With HttpOnly, even if XSS executes, document.cookie cannot read the session cookie. The most common theft vector is eliminated.

Why SameSite=Strict matters: Browser won't send the session cookie on any cross-origin request — blocks CSRF and some hijacking scenarios simultaneously.

Why IP binding matters: Even if a token is stolen, a different IP triggers session invalidation. The attacker's replay attempt is blocked.

Real-world impact: Session hijacking is in OWASP Top 10 (Broken Authentication). High-profile examples: Firesheep (2010) tool demonstrated mass session stealing on public WiFi.
