WHY COOKIE-BASED ROLES ARE DANGEROUS

What is happening: In insecure mode, the server reads the role from a cookie the client sends. Cookies are user-controlled — anyone can open browser dev tools and change them.

Attack flow: User logs in as "demo" (user role) → opens console → runs document.cookie="role=admin" → refreshes → server reads role=admin from cookie → full admin access granted!

Why it's widespread: Developers sometimes use cookies to "remember" the role to avoid DB lookups. This is a Broken Access Control vulnerability — OWASP Top 10 #1.

Secure fix: Role is stored in the signed, server-side session (Flask's SecureCookie uses HMAC — client cannot forge it). Any privilege change requires a DB update by an admin.

Real-world examples: Numerous early SaaS apps had this pattern. The "isAdmin=true" cookie is a classic penetration testing finding.
