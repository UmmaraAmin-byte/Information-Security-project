CSRF ATTACK FLOW
ATTACK SCENARIO
  1. Victim logs into banking site (session cookie set)
  2. Victim visits attacker's page
  3. Attacker's hidden form auto-submits to banking site
  4. Browser sends victim's cookie automatically
  5. Without CSRF check — bank executes the transaction!
CSRF TOKEN DEFENSE
  1. Server generates unique random token per session
  2. Token embedded in every form as hidden field
  3. On POST, server validates token matches session
  4. Attacker cannot read token from another origin (SOP)
  5. Forged request lacks valid token → rejected
REAL-WORLD IMPACT

Real attacks: Netflix (2006) — account hijacking via CSRF. Twitter (2009) — forced follows. eBay — account data changes.

Additional defenses: SameSite=Strict cookie prevents the browser sending cookies on cross-origin requests at all. Double Submit Cookie pattern. Origin/Referer header checking.

Why it's sneaky: The victim doesn't need to click anything — visiting a page with a hidden auto-submit form is enough.
